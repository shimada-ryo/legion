## Lessons

1. **Plan-vs-reality cross-check is non-negotiable.** Three plan-vs-reality mismatches surfaced in a02 alone:
   - `ThreadItem` discriminator field is `type`, not `item_type` (plan was wrong; verified against `node_modules/@openai/codex-sdk/dist/index.d.ts`).
   - `ThreadErrorEvent.message` is top-level; `TurnFailedEvent.error.message` is nested. Plan conflated them. Caught in spec review; fixed in `97792fe`.
   - `AgentEventType` union didn't include `assistant_message` / `session_end` (which Phase 2 Claude provider already implicitly uses through `event-convert.ts`). Extended the union in core types as part of Task 3.

   Pattern to keep: read the actual `.d.ts` / source / interface before blindly applying plan snippets.

2. **`tool_use` vs `tool_call`.** Plan a02 used `tool_use` as an `AgentEvent.type`. Real codebase convention (`event-convert.ts`) translates SDK `tool_use` blocks to `AgentEvent.type='tool_call'`. Implementer correctly aligned to existing convention. Pattern: when porting plan code that mentions a value name, grep the codebase first to see if a different name is established.

3. **Mid-task refactoring beats post-hoc cleanup.** a01 Task 7 ended with `handle()` at 106 lines (over the 100-line policy). I dispatched a refactor pass that split it into `resolveSpawnInputs()` + `runSpawnedAgent()` BEFORE Task 8/9 added more code. Final `handle()` is 55 lines. If the refactor had been deferred to "the end", it would have had to untangle 3 tasks worth of additions simultaneously.

4. **Dead code from earlier phases is real friction, not a no-op.** The Phase 2 channel-based blackboard types (`BlackboardChannelMessage<T>`, etc.) had zero callers but their existence in `blackboard.ts` made the Phase 3 type addition awkward (name collision with new `BlackboardMessage`). First implementer attempt was to rename the old ones to `BlackboardChannelMessage` and keep them — CLAUDE.md flags this as a backwards-compat hack. Spec reviewer caught it; second pass deleted them clean. Pattern: when adding to a file that has stale types, delete first, add second.

5. **The implementer's test-count reporting was repeatedly wrong.** Three different subagent runs reported "13 web failures" or "172 pass / 1 skip" or similar. The actual state was always green. Cause unclear — possibly a stale terminal buffer or subagent confusion about running tests from a non-root cwd. Pattern to apply: always re-verify test counts from main session before trusting them.

6. **Commit subject `1c5212c` (82 chars) was the only convention violation this session.** Above the 70-char limit. Caught in code review but not rebased (cost vs benefit; main is not yet pushed, so a rebase is still cheap if you want a clean history). All subsequent commits were under 70 chars after I explicitly told each implementer to verify.
