### Prerequisites

- **Anthropic auth ‚ÄĒ one of the following.** The Agent SDK resolves in this priority order:
  1. `ANTHROPIC_API_KEY` (Anthropic API contract / Console credits) ‚ÄĒ `sk-ant-api03-...`
  2. `CLAUDE_CODE_OAUTH_TOKEN` (Claude Code Pro/Max subscription, long-lived 1-year token from `claude setup-token`) ‚ÄĒ `sk-ant-oat01-...`
  3. `%USERPROFILE%\.claude\.credentials.json` on Windows / `~/.claude/.credentials.json` on Linux/macOS (cached from interactive `claude /login`) ‚ÄĒ short-lived
  
  **Important:** `ANTHROPIC_API_KEY` silently overrides the lower-priority methods if set. If you intend to use a subscription, verify `ANTHROPIC_API_KEY` is unset first (`echo $ANTHROPIC_API_KEY` / `$env:ANTHROPIC_API_KEY`). For subscription-only users without API credits, an accidentally-set `ANTHROPIC_API_KEY` will cause auth failures.
  
  Recommended setup for subscription users:

  ```bash
  # one-time, generates a 1-year token printed to stdout
  claude setup-token
  # then export the token (or write it to a .env file Bun will auto-load)
  export CLAUDE_CODE_OAUTH_TOKEN=sk-ant-oat01-...
  ```

  PowerShell equivalent:

  ```powershell
  claude setup-token
  $env:CLAUDE_CODE_OAUTH_TOKEN = "sk-ant-oat01-..."
  ```

  Or just rely on `~/.claude/.credentials.json` from a fresh `claude /login` if you don't mind re-auth every few hours.

- `git` on PATH (Phase 1 spawns subprocesses for worktree ops)
- Bun 1.3.14 or later
- A test repository the agent will edit. **Do NOT use the legion repo itself for this** ‚ÄĒ the agent will create worktrees of whatever repo you launch the server from, and Implementer agents commit to branches. Pick a small scratch repo or initialize a fresh one:

  ```bash
  mkdir /tmp/legion-smoke && cd /tmp/legion-smoke
  git init -b main
  git config user.email you@example.com
  git config user.name you
  echo "# smoke" > README.md
  git add README.md
  git commit -m initial
  cp <path-to-legion>/workflows /tmp/legion-smoke/ -r  # bring workflow definitions
  ```

  Or run the server against any existing repo of yours that has a `workflows/` directory copied in.

In one terminal, from the **smoke repo** (not the legion repo):

```bash

ANTHROPIC_API_KEY=sk-... bun run --cwd <legion-repo>/packages/server start
```

Or, equivalent in two steps:

```bash
cd <legion-repo>

# But this uses legion's own cwd ‚ÄĒ the worktrees go to ~/.legion/worktrees/<legion-fp>/...

**Windows note:** invoking `start.ts` directly is shorter and stops more reliably on Ctrl-C than the `bun run --filter`/`--cwd` chain (which deepens the process tree and breaks SIGINT propagation on Windows). ```powershell

```

If the server won't die on Ctrl-C, kill by port: see the Step 4 Windows note for the `Get-NetTCPConnection` snippet (replace `5173` with `5500`).

### Common pitfalls to flag

- **Vite proxy hits 5500 hard-coded.** If you change the legion server port, edit `packages/web/vite.config.ts`.
- **Agents commit on a branch but never push.** That's by Phase 1 design (no PR integration yet). Branches live in your scratch repo.
- **Worktrees pile up** under `~/.legion/worktrees/<repo-fp>/<wfId>/<agentId>/`. `legion cleanup` is the way to clean them.
- **`describe.skipIf` may not skip on some Bun versions.** If the integration test runs unexpectedly without an API key, you'll see a real Anthropic call attempt ‚ÄĒ not a problem, but unexpected.
- **`feature-implementation.yaml` Director can't write.** Expected. Use `bug-fix.yaml` for live edit smoke.
- **WS race on history-then-tail.** Events emitted between the `history()` call and `tail()` subscription can be dropped. With Phase 1's single-agent flow this is rarely visible; flag it if any UI lag appears.

## Phase 2 enablement concerns

The current shape works for Phase 1 single-role spawning but has known seams for Phase 2:

1. **`ctx.adapters` map keyed by `workflowInstanceId`** ‚ÄĒ fine for 1 workflow = 1 agent. Phase 2's Director‚ÄďWorker means many agents per workflow; re-key by `sessionId` so `POST /instances/:id/approvals/:approvalId` resolves to the right adapter.
2. **`WorkflowInstance.agentInstances: []` is always empty.** Need an `agent_instances` SQLite table joined to workflow_instances, populated as `triggerWorkflow` (and Phase 2's delegate spawn) creates agents.
3. **`worktreeAdd` only accepts a commit SHA**, not a branch ref. Phase 2 Reviewer worktrees should `--detach <branch>` (so Reviewer sees the latest Implementer commit, not a frozen base). Extend the wrapper.
4. **`triggerWorkflow` spawns only the role at the trigger.** Phase 2's `delegates` edge needs a real graph walk. Currently `firstRoleNode` has a doc-order fallback that won't generalize.
5. **`CanvasOverlay.deriveActiveRoles`** highlights all role nodes once any event arrives. Phase 2 needs a `sessionId ‚Üí roleNodeId` map, which depends on (2).
