## What was produced

### Trigger UI on TemplateDetail

[packages/web/src/pages/TemplateDetail.tsx](../../../packages/web/src/pages/TemplateDetail.tsx) +
[packages/web/test/pages/TemplateDetail.test.tsx](../../../packages/web/test/pages/TemplateDetail.test.tsx).

The web UI has had no "Trigger workflow" control since Phase 1; every
manual E2E went through `Invoke-RestMethod` against the backend route
that has existed all along. Added an always-visible panel under the
TemplateDetail header: `userPrompt` textarea (3 rows, required — empty
submit emits an inline warn instead of firing the request), `baseRef`
text input (placeholder `HEAD`, blank input is omitted from the request
body so the server falls back to `HEAD`), and a Trigger button that
disables itself while submitting. Success navigates to
`/instances/<id>` (existing InstanceDetail page subscribes to WS and
picks up live events); server failure shows the error inline and
re-enables the button for retry. Dropped the now-misleading
"(read-only mockup — Phase 1 では編集不可)" notice. Tests cover form
render, empty-submit warn, success path with navigation, baseRef
omission, and the server-error path.

### Reviewer retry surfaced on the canvas and the Overview tab

[packages/web/src/components/CanvasOverlay.tsx](../../../packages/web/src/components/CanvasOverlay.tsx),
[packages/web/src/components/sidebar-tabs/OverviewTab.tsx](../../../packages/web/src/components/sidebar-tabs/OverviewTab.tsx),
[packages/web/src/pages/InstanceDetail.tsx](../../../packages/web/src/pages/InstanceDetail.tsx) wiring,
plus tests.

Phase 3 emitted `system.review.decision` Blackboard messages but the UI
had no surface for retry. Two cheap surfacings:

- **CanvasOverlay**: for any role node with multiple agent_instances
  render a `×N runs` hint inside the node; for reviewer roles also
  render a colored chip showing the **latest** decision. The chip uses
  `--status-success` / `--status-warning` / `--status-error`, so the
  retry/approve outcome reads from the canvas before you click anything.
- **OverviewTab**: sort agent_instances by `startedAt` ascending and
  prefix each with `Run #N of M` plus the same colored chip in the
  header. The existing detail block (Decision / Feedback / Notes) is
  preserved under each header.

`InstanceDetail.tsx` threads `blackboardMessages` through to
`CanvasOverlay`. Tests cover both surfaces with a non-reviewer control
case (no chip on Implementer / Director).

User verbatim sentiment from session 2 was 「正直地味すぎるけどね」;
this is the cheapest visible step away from that. Bigger UX moves
(timeline tab, per-instance ephemeral canvas nodes, inline diff
annotations of Reviewer feedback) remain Phase 4 candidates.

## Decisions surfaced this session

1. **Brainstorming-skill discipline survives even on trivial features.**
   The Trigger UI work was ~70 LoC, but going through "what placement,
   what fields, what scope, what tests" before coding saved revisiting
   choices mid-implementation and produced a cleaner commit. Skip the
   written spec doc for changes this small, keep the design proposal
   in the chat.

2. **The `×N` mental model:** the canvas shows **role nodes** (template
   level), not **agent_instances** (runtime level). The session needed
   an explicit user-facing reconciliation of these two layers before
   choosing the retry visualization — without it the user would have
   reasonably expected the reviewer node to literally split into two
   on retry (option D, "per-instance ephemeral nodes"). Recording this
   here in case future visualization work has to settle the same
   distinction again. The chosen surfacing (count + chip inside the
   single role node) is option A+B from that conversation.

## Phase 4 scope candidates (refined)

The retry surfacing here is option A+B from the earlier brainstorm;
**options C and D are still on the table for Phase 4:**

- **C — Timeline / swimlane tab.** Vertical time axis with one
  swimlane per agent_instance. retry, parallelism, and overall workflow
  shape become visible without clicking. Estimated half a day plus a
  short spec.
- **D — Per-instance ephemeral nodes on the canvas.** The user's
  intuition is "2 reviewers ran ⇒ 2 boxes on the canvas". Doing this
  cleanly requires layout work (where to place child nodes, how to
  fade ones that have ended, how to interact with template-level
  layout) and will collide with whatever the design-1 worktree is doing
  with canvas drag. 1-2 days plus design coordination.

Other Phase 4 candidates from prior sessions remain:

- Reviewer feedback as inline diff annotations.
- Per-topic Blackboard swimlanes instead of a flat tab.
- Agent thinking time / token usage indicator.
- D-014 reactor / Inbox.
- Multi-Implementer parallelism.
- Codex-as-Implementer with approval flow.
- Docker sandbox (anchors a few of the above).
