### Bug-fix wave (real-SDK debug discoveries)

These are the bugs the integration tests surfaced. Each one would have shipped
broken without the real-SDK gate. CLAUDE.md "Test Policy: Mocks Require Contract
Tests" paid for itself many times today.

1. **`053be71`** — `REVIEW_OUTPUT_SCHEMA` was missing `additionalProperties: false`.
   The Codex contract test caught this first: OpenAI's strict response_format
   rejects schemas without it (`invalid_json_schema`). Production code used the
   exact same broken shape.
2. **`5be0793`** — OpenAI strict mode also requires *every* property be in
   `required`. Phase 3 had `required: ['decision']` only; optional `feedback` and
   `notes` triggered another `invalid_json_schema`. Fix: list all three in
   `required` and switch their types to `['string', 'null']` (nullable union).
   `parseReviewerOutput` already tolerates `notes ?? ''`; the
   DelegateToolOutput guard was tightened from `!== undefined` to
   `typeof === 'string'` for `feedback`.
3. **`cf65c02`** — `runSpawnedAgent` only captured `evt.payload.text` from
   `evt.type === 'message'`. Codex Reviewer emits `type === 'assistant_message'`
   with `payload.content`. Result: Reviewer summary was always empty →
   `parseReviewerOutput('')` returned no payload → no decision parsed → no
   Blackboard decision event → Implementer was given `decision: undefined` and
   either hung or ended without a proper answer. Fixed by accepting both shapes
   (`payload.text ?? payload.content`).
4. **`08745fc`** — `provider.launch` in `runSpawnedAgent` did NOT receive the
   legion MCP server, so the Implementer had no `delegate` tool exposed and
   could not self-delegate to the Reviewer. The trigger.ts code passed
   mcpServers ONLY for the Director. Fix: a recursive `legionMcpFactory` closure
   that, given a parent agent ID, builds a fresh `DelegateToolHandler` (scoped
   to that ID) and wraps it in a fresh MCP server. Threaded through
   `DelegateToolDeps.legionMcpFactory` (optional; mock-friendly).
5. **`8a851c2`** — Implementer agent attempted `mkdir -p src/` when `src/`
   didn't exist in scratch-repo. `mkdir` is not in `defaultAllowedToolsFor('implementer')`,
   triggering `permission_request` with no human in the loop → hang. Workaround
   for the test: pre-create `src/`. **Underlying issue still open — see "Open
   issues".**
6. **`b0f6181`** — IMPLEMENTER_PROMPT said "if a reviewer is wired up" — the LLM
   read this as optional and skipped the delegate(reviewer) call. Rewrote as
   imperative ("you MUST attempt to invoke the Reviewer via the legion delegate
   tool"). Unit test phrase updated accordingly.
7. **`cf65c02` + `c607bc6`** — Added `packages/runtime/src/util/logger.ts`:
   stderr-only debug logger gated by `LEGION_DEBUG=1`. Strategic log points at
   `trigger.start`, `drainStream.statusChange`, `delegate.handle.enter/spawn/drained/reviewerParse`,
   `runSpawned.launch.start/done`, `codex.launch`, `codex.stream.event/done`.
   Also bumped integration timeouts (240→500s for approve, 360→740s for retry)
   and made `dumpStateOnFailure` fire after every test run. The error-channel
   field in the log keeps up to 600 chars (other strings 200) so OpenAI-rejection
   payloads are legible.

## Lessons

1. **Real-SDK testing is non-negotiable when shipping a new provider.** Today's
   debug wave found 6 real bugs in a row that all unit tests passed through
   because they mocked the provider boundary. The two contract tests in a02 +
   the Phase 3 E2E in a06 collectively cost ~$3-5 in real API fees but caught
   ~3 days of would-be debugging in production.
2. **OpenAI strict response_format JSON Schema is fussy and undocumented in the
   places you read first.** The two non-obvious rules:
   - `additionalProperties: false` is required (not "recommended")
   - `required` must include every key in `properties` (so to model optional
     fields, use `type: ['string', 'null']`)
   Both surface as `invalid_json_schema` with quite readable detail in
   `event.error.message`, but the SDK's raw event field is `event.error`
   (an object) for `turn.failed` and `event.message` (a string) for `error`.
3. **Spawned agents need their own MCP server, not the parent's.** Each agent
   needs a `delegate` tool bound to a handler with `parentAgentInstanceId = self`,
   so the graph-walker resolves the right outgoing edges. Previously only the
   Director got the MCP server, which is exactly why the Phase 3 Implementer→
   Reviewer flow never worked end-to-end before today.
4. **LLM-level prompt is load-bearing.** Even with all wiring correct, an
   Implementer that reads "if a reviewer is wired up, you have a delegate tool"
   will sometimes decide there is no reviewer and skip the call. Imperative
   prompts ("you MUST attempt to invoke the Reviewer") + an explicit "if you
   get an error indicating the Reviewer is not configured" fallback works
   reliably so far.
5. **`bun.env` loading is per-cwd**: when running tests with `bun test --cwd
   packages/runtime ...`, bun loads `.env` from `packages/runtime/`, not the
   repo root. This session works around it by reading repo-root `.env` in
   PowerShell and exporting vars before invoking bun. Worth wrapping into a
   small `with-env.ps1` helper in a future session.

## Open issues (new in this session)

### P1 — blocking `phase3-end`

- **Bash whitelist is too narrow for real Implementer behaviour.** The role
  profile lists only specific test/build/git commands. Real Claude agents
  routinely use `cd`, `ls`, `pwd`, `mkdir`, `cat`, `git rev-parse`, etc. Without
  them whitelisted (or auto-approved), the workflow hangs on permission_request
  in non-interactive contexts (and shows confusing UI noise in interactive use).
  Two paths:
  - **(A)** Extend `IMPLEMENTER_BASH_WHITELIST` with a curated list of common
    read-only / no-side-effect commands: `cd*`, `ls*`, `pwd*`, `cat*`, `mkdir*`,
    `git log*`, `git rev-parse*`, `git branch*`, `git show*`.
  - **(B)** Add an auto-approver to `ApprovalOrchestrator` that approves any
    command matching a "safe-prefix" allowlist when the SDK consults `canUseTool`.
    Cleaner but a bigger change.
  Recommendation: ship (A) as a quick fix to land Phase 3, leave (B) as Phase 3.5
  cleanup.

- **Retry E2E flakiness:** the Codex Reviewer sometimes approves on the first
  pass even with the prompt explicitly engineered to provoke `request-changes`.
  This is fundamentally LLM-driven; treat the retry test as "best-effort, expect
  occasional first-round approves." After (A) above lands, we should also weaken
  the assertion to: "either retry happened (≥2 reviewers, last decision approve)
  OR a single-round approve completed cleanly." A workflow that approves faster
  than expected is not a bug.

### P2 — drive-by improvements surfaced today

- **`runSpawnedAgent` ignores `session_end` events with `status: 'failed'`.** When
  a spawned agent's provider stream ends with a failed session_end event, the
  local `status` variable stays `'completed'`. We log it but don't reflect it
  in the DelegateToolOutput. Should distinguish "spawn errored in provider"
  from "spawn errored in legion try-catch".
- **`AgentEvent.type` union still doesn't include `'blackboard.message'`.** The
  WS wrapper uses a synthetic top-level field (`{type, message}`) that is NOT
  an AgentEvent — clients discriminate, but if any future code does
  exhaustive switch over AgentEvent it should also know the WS produces this
  envelope. Worth a JSDoc on `useInstanceEventStream` and/or a shared union
  type for WS payloads.
