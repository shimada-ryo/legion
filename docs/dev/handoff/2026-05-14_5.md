## Branch policy (READ CAREFULLY — policy changed this session)

**`--no-ff` policy is REPEALED. legion now uses linear history (fast-forward merge) as the default.**

- Use `git merge --ff-only <branch>` (or simply `git merge <branch>` — it will ff if possible).
- If `main` has advanced since the feature branch diverged, **rebase the feature branch onto `main`** to restore linearity, then ff-merge. Do NOT fall back to `--no-ff`.
- **Phase boundaries are recorded with annotated tags**, not merge commits. Convention: `phase<N>-<scope>-end`. Example just created: `phase2-narrow-end`.
- The Phase 1 `--no-ff` merge commit (`07ced0d`) is left untouched as a historical artifact of the old policy.
- The `2026-05-14_4.md` handoff still references the old `--no-ff` policy. That doc is frozen as written; this handoff supersedes it for branch policy.

Why the change: improved `git log` readability, clean `git bisect`, single-column timeline. The trade-off (loss of merge commits as phase markers) is covered by tags.

### Critical bug found by the integration test (and fixed: `c4d043a`)

The test failed on first auth-run with `agent_instances.length === 1` — the Implementer was never spawned. Diagnosis:

- `packages/runtime/src/orchestrator/trigger.ts` was constructing `customTools` as a plain `[{name, description, inputSchema (JSON Schema), handler}]` array.
- `packages/runtime/src/adapter/provider/launch.ts` was passing it as `tools: req.customTools` to the SDK's `query()`.
- **The SDK has no `tools:` option.** It silently ignored the field, so the Director was launched with allowedTools containing `'mcp__legion__delegate'` but no actual tool registered under that name. The Director ran, did some Reads, concluded it had no editing capability, and ended.
- Fix (`c4d043a`): use the SDK's intended API — `tool('delegate', ..., { role: z.string(), prompt: z.string(), rationale: z.string().optional() }, handler)` + `createSdkMcpServer({ name: 'legion', tools: [delegateTool] })` + pass via `mcpServers: { legion: server }`. Added `zod` to `@legion/runtime` deps. The `LaunchRequest.customTools: unknown[]` field on `@legion/core/types/agent-provider.ts` was replaced with `mcpServers?: Record<string, unknown>`.
- Tool name format `mcp__<server-name>__<tool-name>` auto-generates to `mcp__legion__delegate`, which already matched `role-profile.ts` allowedTools.

This is the kind of bug **only a real-SDK integration test can catch** — unit tests mock `query` and accept any input shape. The 38.7s + ~10 cents of API budget was money well spent.

## Lessons (carry into Phase 3)

1. **Verify before commit.** During session 5 the wiring fix was committed (`c4d043a`) before the integration test was re-run with auth — it worked out (test passed) but it was a process slip. The user called this out. **Pattern to keep**: when the validating signal is downstream of the change (e.g., requires user-side env or external API), do not wire `commit` into the implementer subagent prompt. Apply changes, verify, then commit as a separate step.

2. **Plan correctness is provisional.** The a04 plan had 3 bugs (non-existent function, wrong concurrency model, missing model knob). All were caught at implementation time, not at plan-write time. Implementers (and the controller) should always cross-check plan code against the actual API surface before starting.

3. **`unknown[]` types hide API mismatches.** `LaunchRequest.customTools: unknown[]` had zero compile-time signal that the entire shape was wrong. Where the boundary crosses out of legion's control (e.g., into the SDK), prefer concrete types (or at least `Record<string, unknown>` keyed by an SDK-recognized name) over `unknown[]`.

4. **xyflow's `data` field does NOT reach the DOM.** When a test needs to assert on rendered attributes, use a custom `nodeTypes` renderer. (Documented in `CanvasOverlay.tsx` with a comment.)

5. **`bun run test` not `bun test` from repo root.** Running root-level `bun test` (without `run`) loads .tsx files outside the workspace runner and yields spurious `document is not defined` failures. Always `bun run test`. Implementer subagents have been wrong about this twice; controller must re-verify.
