## Lessons

1. **`@openai/codex-sdk` exists — `subprocess` was a premature assumption.**
   Initial framing in §D of the spec assumed Codex would require hand-rolled
   `codex exec` subprocess invocation. A grounding-skill investigation (dual-agent
   Researcher + Verifier) confirmed Codex ships a TypeScript SDK at
   `@openai/codex-sdk` (v0.130+), symmetric to `@anthropic-ai/claude-agent-sdk`.
   Internally the SDK still spawns `codex exec --experimental-json` — but so does
   Claude Agent SDK internally. The user explicitly pushed back ("Claude Code のときも
   subprocessで最初構想していたが、仕様理解不十分だったためライブラリを使うことになった。codex
   も同じことになるのでは。") — they were right. Spec §D was rewritten before
   commit. Pattern to keep: **whenever assuming "we'll subprocess this," check
   for an official in-process SDK first.**

2. **"narrow" suffix is meaningless padding for Phase 3.**
   I initially copied Phase 2's "Phase 2 narrow scope" vocabulary into the Phase 3
   spec, but Phase 2's `narrow` had real structural meaning (D-036 cut 4 of 5
   originally-planned subsystems). Phase 3 only defers one item (Workflow Editor
   editing → Phase 3.5), which is just normal phase boundary work, not a "narrowing."
   The user called this out ("narrowっていらなくない？") — correct. Removed
   inline. Tag is `phase3-end` (no `narrow` suffix). Phase 2's `phase2-narrow-end`
   stays as a historical artifact. **Pattern to keep**: don't mechanically copy
   prior-phase terminology. Each phase's vocabulary should earn its place.

3. **Plan self-review caught a real gap.**
   First draft of a02 finished without a task for `DelegateToolHandler` to do
   **dynamic provider selection** (target node's `provider` field → `providers: Map`
   lookup). a01 used a single-provider stub. Without dynamic selection, a02's
   Codex provider couldn't actually be reached at runtime. Added a02 Task 6
   (provider dynamic selection) before commit. Renumbered Task 6/7 → 7/8.
   **Pattern to keep**: after writing a plan, ask "could the next agent
   actually execute this end-to-end?" not just "does each task make sense?"

4. **CLAUDE.md is the right home for project-global policies, not specs.**
   Initial discussion of the Mocks Require Contract Tests rule was happening
   inside Phase 3 spec §G. User pointed out this is a project-global rule
   that should live in CLAUDE.md so it survives across phases. Moved it.
   Spec §G now references CLAUDE.md and only lists the Phase 3-specific
   mock↔contract pairings. **Pattern to keep**: if the rule applies to all
   future phases, CLAUDE.md is the canonical home; phase-specific specs
   reference it.
